WaferFlow Spec

This page is the formal specification for WaferFlow v0.1.0. For an introduction, see WaferFlow Overview.

Types

WaferFlow

The top-level flow definition.

{
    "id":          string,         // required — unique flow identifier
    "name":        string,         // required — human-readable name
    "version":     string,         // required — semver (e.g. "1.0.0")
    "description": string | null,  // optional
    "input":       PortSchema | null,  // optional — schema for $.input
    "output":      PortSchema | null,  // optional — schema for flow output
    "steps":       Step[],         // required — at least one step
    "config":      FlowConfig | null   // optional
}

Step

A single execution unit in the flow.

{
    "id":          string,              // required — unique within flow, pattern: [a-zA-Z][a-zA-Z0-9_-]*
    "block":       string,              // required — block name to execute
    "input":       object | null,       // optional — input map, values can be expressions
    "next":        NextEntry[] | null,  // optional — conditional routing
    "each":        string | null,       // optional — expression resolving to array
    "parallel":    ParallelBranch[] | null,  // optional — concurrent branches
    "description": string | null        // optional
}

Fields are mutually exclusive where it makes sense: a step with each iterates, a step with parallel fans out, and a step with next routes conditionally after execution.

NextEntry

A conditional routing entry. Evaluated in order; the first match wins.

{
    "when": string | null,  // optional — expression; omit for default route
    "step": string | null,  // target step id (exactly one of step/flow required)
    "flow": string | null   // target flow id (exactly one of step/flow required)
}

If when is omitted, the entry is the default (fallback) route. If no entry matches and there is no default, execution continues to the next step in the array.

ParallelBranch

A branch in a parallel step. All branches execute concurrently.

{
    "steps": Step[]  // required — at least one step per branch
}

Results from all branches are merged into the accumulator. Each branch's steps write to the same shared accumulator, so step ids must be unique across all branches.

PortSchema

A JSON Schema subset for typed ports.

{
    "type":        string | null,       // "object", "array", "string", "number", "integer", "boolean", "null"
    "properties":  { [key]: PortSchema } | null,
    "items":       PortSchema | null,   // for array types
    "required":    string[] | null,
    "description": string | null,
    "default":     any | null
}

FlowConfig

Flow-level execution settings.

{
    "timeout_ms": integer | null,  // max execution time in milliseconds
    "max_steps":  integer | null,  // max step executions (prevents infinite loops)
    "on_error":   "stop" | "skip" | "retry"  // default: "stop"
}
  • stop — abort the flow on any step error
  • skip — skip the failed step and continue to the next
  • retry — retry the failed step (implementation-defined retry policy)

BlockDef

Block metadata. Defined separately from flows in a block.json file.

{
    "id":          string,              // required — e.g. "@wafer-run/wafer-run/sqlite"
    "name":        string,              // required — human-readable name
    "version":     string,              // required — semver
    "description": string | null,       // optional
    "input":       PortSchema | null,   // optional — input contract
    "output":      PortSchema | null,   // optional — output contract
    "runtime":     "wasm" | "native" | "both" | null  // optional
}

Flow Control

Sequential (default)

Steps execute in array order. Each step's output is added to the accumulator before the next step runs.

"steps": [
    { "id": "step-a", "block": "block-a" },
    { "id": "step-b", "block": "block-b", "input": { "data": "$.step-a.result" } },
    { "id": "step-c", "block": "block-c", "input": { "data": "$.step-b.result" } }
]

Conditional Routing (next)

After a step executes, its next array is evaluated. The first NextEntry whose when expression is true (or has no when) determines the next step.

{
    "id": "check",
    "block": "validator",
    "next": [
        { "when": "$.check.valid == true", "step": "process" },
        { "step": "reject" }
    ]
}

If a next entry targets a step, execution jumps to that step id. If it targets a flow, execution transfers to a different flow entirely.

Iteration (each)

When a step has each, it runs once per item in the resolved array. During each iteration, $.each.item is the current element and $.each.index is the zero-based index.

{
    "id": "process-items",
    "block": "item-processor",
    "each": "$.load-data.items",
    "input": {
        "item": "$.each.item",
        "position": "$.each.index"
    }
}

The step's output in the accumulator is an array of all iteration results.

Parallel Branches

When a step has parallel, all branches execute concurrently. Each branch is an independent sequence of steps.

{
    "id": "fan-out",
    "block": "noop",
    "parallel": [
        {
            "steps": [
                { "id": "branch-a", "block": "fetch-api-a" }
            ]
        },
        {
            "steps": [
                { "id": "branch-b", "block": "fetch-api-b" }
            ]
        }
    ]
}

All branch step outputs are merged into the accumulator. Execution continues with the next step in the main array after all branches complete.

Expression Syntax

Expressions are used in step input values and when conditions.

Path Expressions

A path starts with $. and traverses the accumulator using dot-separated segments:

  • $.input — the flow's input data
  • $.input.email — a field on the input
  • $.find-user.name — output of step find-user, field name
  • $.step1.items.0 — array index access
  • $.step1.data.nested.value — deep nesting
  • $.each.item — current iteration item (inside each loops)
  • $.each.index — current iteration index (inside each loops)

Literals

  • Strings: "hello" or 'hello'
  • Numbers: 42, 3.14
  • Booleans: true, false
  • Null: null
  • Arrays: ["a", "b", "c"] (JSON syntax, used with in/not in)

Comparison Operators

Used in when conditions. Both sides can be paths or literals.

  • == — equal (deep JSON equality)
  • != — not equal
  • > — greater than (numeric)
  • < — less than (numeric)
  • >= — greater than or equal (numeric)
  • <= — less than or equal (numeric)

Examples:

$.check.match == true
$.step1.count > 5
$.input.email != null

Logical Operators

  • && — logical AND (short-circuiting)
  • || — logical OR (short-circuiting)

Example:

$.input.active == true && $.check.count > 0

Membership Operators

  • in — value is contained in an array
  • not in — value is not contained in an array

The right side must resolve to an array.

"admin" in $.find-user.roles
$.input.status not in ["banned", "suspended"]

Accumulator Pattern

The accumulator is a flat key-value store where keys are step ids and values are step outputs (JSON). It is built up during execution:

  1. Before the first step, $.input is set to the flow's input data.
  2. After each step executes, its output is stored as $.{step-id}.
  3. Step input values that are expression strings ("$.foo.bar") are resolved against the accumulator before the block runs.
  4. For each steps, $.each.item and $.each.index are set for each iteration.
  5. For parallel steps, all branch outputs are merged into the same accumulator.

Non-expression strings and other JSON values in step inputs are passed through as-is.

Error Handling

The config.on_error field controls what happens when a step fails:

  • stop (default) — the flow aborts immediately and returns the error.
  • skip — the failed step is skipped; its output is not written to the accumulator. Execution continues with the next step.
  • retry — the runtime retries the failed step. The retry policy (count, backoff) is implementation-defined.

The max_steps config prevents infinite loops from next routing that cycles back to earlier steps. When the limit is reached, the flow aborts with an error.
