WaferFlow

WaferFlow is a declarative, medium-agnostic flow specification. It describes how data moves through a sequence of steps, where each step executes a block and its output is stored in a shared accumulator. Downstream steps reference previous outputs using $.step-id.field expressions.

Design Philosophy

WaferFlow is built on three principles:

  • Blocks are pure functions — each block receives input and produces output. No implicit shared state between steps.
  • Data flows through an accumulator — every step's output is stored under its id. Later steps reference earlier outputs with $.step-id.field path expressions.
  • The spec is standalone — WaferFlow is not tied to wafer-run, HTTP, or any specific runtime. It works for APIs, interactive content, IoT pipelines, form wizards, or anything that can be modeled as data flowing through steps.

Core Concepts

  • Flow — the top-level definition: an id, name, version, ordered list of steps, and optional input/output schemas.
  • Step — a single unit of work. Each step has an id, references a block, and optionally defines its input mapping, next routing, each iteration, or parallel branches.
  • Block — the executable logic behind a step. Identified by name (e.g. @wafer-run/wafer-run/sqlite). Blocks are defined separately from flows and can be reused across many flows.
  • Accumulator — a key-value store that collects step outputs during execution. The flow's input is available as $.input. Each step's output is available as $.step-id.
  • Expressions — $.step-id.field path references used in step inputs and routing conditions. Supports comparisons (==, !=, >, <), logical operators (&&, ||), and membership tests (in, not in).

Quick Example

A simple 3-step flow: look up a user, check their role, and return a greeting.

{
    "id": "greet-user",
    "name": "Greet User",
    "version": "1.0.0",
    "input": {
        "type": "object",
        "properties": {
            "user_id": { "type": "string" }
        },
        "required": ["user_id"]
    },
    "steps": [
        {
            "id": "find-user",
            "block": "@wafer-run/wafer-run/sqlite",
            "input": {
                "action": "get",
                "collection": "users",
                "id": "$.input.user_id"
            }
        },
        {
            "id": "check-role",
            "block": "@app/role-checker",
            "input": {
                "roles": "$.find-user.roles"
            }
        },
        {
            "id": "greet",
            "block": "@app/greeter",
            "input": {
                "name": "$.find-user.name",
                "is_admin": "$.check-role.is_admin"
            }
        }
    ]
}

Steps execute sequentially. The accumulator after all three steps contains $.input, $.find-user, $.check-role, and $.greet. The last step's output becomes the flow's result.

Use Cases

Because WaferFlow is medium-agnostic, the same spec format applies across domains:

  • APIs — request validation, database queries, response formatting
  • Interactive content — rendering scenes, handling user choices, branching narratives
  • IoT — reading sensors, validating ranges, triggering alerts
  • Forms — multi-step wizards with conditional fields and validation

See WaferFlow Examples for complete flows in each domain.
