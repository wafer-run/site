CLI

The wafer CLI is a native Rust binary that scaffolds block projects, builds them to WebAssembly, runs test fixtures, and packages them for deployment. Install it with cargo:

cargo install wafer-cli

wafer new

Scaffold a new block project. The block name must be in {org}/{block} format.

wafer new <name> --lang <rust|go|ts>

Examples

wafer new myorg/hello --lang rust

Creates:

myorg-hello/
  manifest.json        # block metadata
  Cargo.toml           # [lib] crate-type = ["cdylib"]
  src/lib.rs           # starter block with #[wafer_block]
  tests/
    handle_basic.json  # test fixture
wafer new myorg/hello --lang go

Creates:

myorg-hello/
  manifest.json        # block metadata
  go.mod               # module with wafer-sdk-go
  main.go              # starter block with wafer.Export()
  tests/
    handle_basic.json  # test fixture
wafer new myorg/hello --lang ts

Creates:

myorg-hello/
  manifest.json        # block metadata
  package.json         # @wafer-run/sdk dependency
  src/index.ts         # starter block with block()
  tests/
    handle_basic.json  # test fixture

manifest.json

Every block project has a manifest.json at its root. This is the single source of truth for block metadata:

{
  "name": "myorg/hello",
  "version": "0.1.0",
  "interface": "handler@v1",
  "summary": "Returns a hello world response",
  "runtime": "wasm",
  "instance_mode": "singleton",
  "requires": []
}
  • name — two-segment {org}/{block} identifier (e.g. wafer-run/sqlite)
  • version — semver string
  • interface — the interface this block implements (e.g. handler@v1)
  • runtime — always "wasm" for CLI-built blocks
  • instance_mode — singleton, per-flow, per-node, or per-execution
  • requires — list of block names this block may call via call_block()

wafer build

Build the block in the current directory to WebAssembly. Validates the output against the manifest before finishing.

wafer build

The compiler is chosen automatically based on the project layout:

  • Rust — runs cargo build --target wasm32-wasip1 --release, copies output to target/block.wasm
  • Go — runs tinygo build -target wasi -o target/block.wasm .
  • TypeScript — runs the @wafer-run/sdk build tool (WASM compilation coming soon; SDK types are available now)

On success the built artifact is written to target/block.wasm in the project root.

Example output

$ wafer build
   Compiling myorg-hello v0.1.0
    Building wasm32-wasip1 [release]
   Validating target/block.wasm
      Exports malloc, info, handle, lifecycle ... ok
       Manifest name=myorg/hello version=0.1.0 ... ok
      Built target/block.wasm (47 kB)

wafer test

Run test fixtures against the compiled block. Each fixture is a JSON file in the tests/ directory.

wafer test [path]

If path is omitted, all tests/*.json files are run. Pass a specific path to run a single fixture:

wafer test tests/handle_basic.json

Fixture format

{
  "description": "basic hello response",
  "input": {
    "kind": "GET:/",
    "data": null,
    "meta": {}
  },
  "expected": {
    "action": "Respond",
    "status": 200,
    "body": { "hello": "world" }
  }
}

Example output

$ wafer test
  Running tests/handle_basic.json
  Running tests/handle_missing.json
  2 passed, 0 failed

wafer package

Build the block and produce a dist/ directory ready for deployment or publishing to the registry.

wafer package

Output in dist/:

  • block.wasm — compiled WebAssembly artifact
  • manifest.json — block metadata (copied from project root)

The dist/ directory can be uploaded as a GitHub release or placed in a Solobase blocks/ directory.

Example output

$ wafer package
   Building target/block.wasm ... ok
  Packaging dist/block.wasm
  Packaging dist/manifest.json
    Packaged dist/ (47 kB total)

Registry integration

Packaged dist/ directories published as GitHub releases are automatically discoverable by the package registry. The registry detects the block.wasm and manifest.json assets and exposes them for install.

wafer dev

Watches a wafer-run app's source files and restarts the running process on every save. Prints a wafer-aware boot summary on each successful start (block count, flow count, listening address). Use this as your local inner loop instead of manually cargo run-ing after every edit.

wafer dev [--bin NAME] [--release] [--watch PATTERN]... [-- <cargo args>]
--bin <NAME>
Cargo bin target to run. Auto-detected from Cargo.toml if there's exactly one bin.
--release
Pass --release to cargo. Default is dev profile (faster rebuilds).
--watch <PATTERN>
Add an extra path or glob to watch. Repeatable. Default watch list: src/**/*.rs, Cargo.toml, wafer.lock.
--no-default-watch
Disable the default watch list (use only --watch patterns).
--debounce <MS>
File-change debounce window in milliseconds. Default 200.
--kill-timeout <SEC>
SIGTERM → SIGKILL grace period. Default 3.

The boot summary requires the runtime's structured wafer.runtime tracing events (added in Spec 3B). Apps using older runtime versions still work — wafer dev falls back to omitting per-spawn counts.
