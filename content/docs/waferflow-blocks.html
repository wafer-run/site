WaferFlow Blocks

A block is the executable unit behind each step in a WaferFlow. Blocks receive structured input, do work, and return structured output. This page covers how blocks are defined, named, packaged, and integrated with the runtime.

BlockDef

Every block has a BlockDef that describes its metadata and contract. This is typically stored as a block.json file alongside the compiled block.

{
    "id": "@wafer-run/wafer-run/sqlite",
    "name": "SQLite",
    "version": "0.1.0",
    "description": "CRUD operations on SQLite collections",
    "input": {
        "type": "object",
        "properties": {
            "action": { "type": "string", "description": "get, list, create, update, delete" },
            "collection": { "type": "string" },
            "id": { "type": "string" }
        },
        "required": ["action", "collection"]
    },
    "output": {
        "type": "object",
        "properties": {
            "data": { "description": "Query result" },
            "error": { "type": "string" }
        }
    },
    "runtime": "both"
}

Fields:

  • id — unique identifier following the naming convention (see below)
  • name — human-readable display name
  • version — semantic version (e.g. 0.1.0)
  • description — optional summary of what the block does
  • input — optional PortSchema describing the expected input shape
  • output — optional PortSchema describing the output shape
  • runtime — which runtime targets the block supports: wasm, native, or both

The input and output schemas use a JSON Schema subset (PortSchema) that supports type, properties, items, required, description, and default. These schemas enable tooling to validate flow definitions before execution and provide editor autocomplete for step inputs.

Naming Convention

Block ids follow the @{github_owner}/{repo}/{block} format:

  • @wafer-run/wafer-run/sqlite — core SQLite block from the wafer-run repo
  • @wafer-run/wafer-run/s3 — core S3 storage block
  • @suppers-ai/solobase/auth — authentication block from the solobase repo
  • @acme/my-blocks/validator — a third-party block

Ownership is enforced by GitHub permissions — only the repository owner can publish blocks under their namespace. The registry validates this on package submission.

When referencing blocks in flow steps, use the full id:

{
    "id": "find-user",
    "block": "@wafer-run/wafer-run/sqlite",
    "input": { "action": "get", "collection": "users", "id": "$.input.user_id" }
}

Runtime Types

The runtime field in a BlockDef indicates how the block can be executed:

  • native — compiled Rust that runs directly in the host process. Zero overhead, full access to the host. Cannot be distributed as a standalone package (must be compiled in).
  • wasm — compiled to WebAssembly and loaded at runtime. Cross-language (Rust, Go, TypeScript), sandboxed, distributable via the package registry.
  • both — supports both native and WASM compilation. Core @wafer-run/wafer-run/* blocks are typically both, so they work in any deployment context.

The runtime type is detected automatically from the release assets when publishing to the registry — if a .wasm file is present, the block is marked as wasm. If a Rust crate is referenced, it is marked as native. Both assets means both.

Pure Blocks vs Service Blocks

Blocks fall into two categories based on their side effects:

Pure Blocks

Pure blocks are deterministic transformations: given the same input, they always produce the same output. They do not access databases, storage, network, or any external state.

  • Validators — check input structure, format, ranges
  • Transformers — reshape data, map fields, compute values
  • Formatters — render templates, build response bodies

Example:

// Input:  { "price": 100, "tax_rate": 0.08 }
// Output: { "total": 108.0, "tax": 8.0 }

Service Blocks

Service blocks interact with the runtime context to access databases, storage, cryptographic operations, or external APIs. They declare their dependencies in the requires list of their BlockInfo, and the runtime enforces access control.

  • @wafer-run/wafer-run/sqlite — CRUD against SQLite collections
  • @wafer-run/wafer-run/s3 — object storage operations
  • @wafer-run/wafer-run/postgres — PostgreSQL queries
  • @wafer-run/wafer-run/local-storage — local filesystem storage

Service blocks use the context's call_block() to invoke runtime services:

// A service block that reads from the database
fn handle(ctx: Context, msg: Message) -> BlockResult {
    let user = ctx.call_block("wafer-run/database", query_msg)?;
    // ... process user ...
}

WASM Packaging

A WASM block package consists of two files:

  • block.json — the BlockDef metadata described above
  • block.wasm — the compiled WebAssembly component

The .wasm file exports four functions that the runtime calls: malloc, info, handle, and lifecycle. Guest SDKs (TypeScript, Go, Rust) generate these exports automatically. See WASM Blocks for build instructions.

Build and package with the CLI:

# Build the WASM binary
cargo build --target wasm32-wasip1 --release

# The block.json is generated from #[wafer_block] attributes
# or written manually for non-Rust blocks

Package Registry Integration

Published blocks are discoverable through the wafer.run package registry. The registry stores manifests that point to GitHub Release assets:

{
    "name": "acme/validator",
    "summary": "Input validation block",
    "latest": "1.0.0",
    "versions": {
        "1.0.0": {
            "abi": 1,
            "wasm_url": "https://github.com/acme/validator/releases/download/v1.0.0/block.wasm"
        }
    }
}

When a flow references a block that is not registered locally, the runtime resolves it from the registry, downloads the .wasm, validates ABI compatibility, and loads it. See Package Registry for the full publishing workflow.

Creating Blocks for WaferFlow

Any block that conforms to the wafer-run block interface works in WaferFlow. The flow engine maps step inputs to a Message, calls the block's handle(), and stores the result in the accumulator. No special WaferFlow-specific code is needed in the block itself.

For step-by-step instructions on creating blocks, see Creating a Block (native) and WASM Blocks (cross-language).
